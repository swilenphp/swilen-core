Arquitectura de Plugins para Entornos Asíncronos

Esta aplicación interactiva detalla la especificación del contrato de plugins de WordPress. Su objetivo es proporcionar la base arquitectónica necesaria para refactorizar el Core utilizando Swoole, habilitando un procesamiento asíncrono de alto rendimiento compatible con el ecosistema existente, incluyendo WooCommerce.

🔄 El Ciclo de Vida Tradicional vs Asíncrono

Para entender el impacto de Swoole, primero debemos visualizar cómo WordPress procesa una petición estándar. En PHP-FPM, cada petición arranca de cero. En Swoole, la aplicación reside en memoria.

1

Carga del Core

wp-load.php y wp-config.php.

⚠️ En Swoole, esto ocurre una sola vez al iniciar el servidor.

2

Plugins & Temas

Se ejecutan los archivos principales de los plugins (registro de hooks).

3

Init & Query

Acción `init`. Parseo de la URL y ejecución de la query principal de base de datos (`$wp_query`).

4

Template Render

Se incluye el archivo del tema correspondiente y se envía el HTML al cliente.

El Sistema de Hooks

El corazón de la extensibilidad de WordPress. Los plugins interactúan con el Core a través de la API de Plugins, gestionada por la clase `WP_Hook` o arreglos globales (históricamente `$wp_filter`). Este sistema debe ser completamente asíncrono y thread-safe (aislado por petición) en el nuevo entorno.

🎬 Actions (Eventos)

Permiten ejecutar código personalizado en puntos específicos del ciclo de vida. No retornan valores al Core.

  • plugins_loaded

    Se dispara una vez que todos los plugins activos han sido incluidos.

  • init

    Punto principal para inicializar tipos de post, taxonomías y traducciones.

  • wp_enqueue_scripts

    Momento adecuado para añadir CSS y JS al frontend.

🚰 Filters (Modificadores)

Permiten interceptar y modificar datos antes de que WordPress los guarde en BD o los envíe al navegador. Deben retornar un valor.

  • the_content

    Modifica el contenido del post antes de imprimirlo en pantalla.

  • posts_where

    Permite alterar la cláusula WHERE de la consulta SQL principal de WP.

  • woocommerce_product_price

    Específico de WC: Modifica dinámicamente el precio mostrado de un producto.

Frecuencia de Invocación por Petición

En una petición típica (carga de una página), se disparan miles de hooks. Gestionar esta memoria es el mayor desafío en Swoole.

🌍

El Problema del Estado Global

WordPress se diseñó en la era del "Share Nothing" de PHP. Depende fuertemente de variables globales. En un servidor persistente como Swoole, modificar un global en una petición (Corrutina) afectará accidentalmente a otras peticiones concurrentes (Memory Leaks y Data Bleed).

💾

global $wpdb;

Instancia de la base de datos. En Swoole, debe ser reemplazada por un Connection Pool gestionado dinámicamente, evitando bloqueos de I/O.

👤

global $current_user;

Información del usuario autenticado. Debe extraerse a un Contexto de Petición (Request Context) local a la corrutina actual usando Swoole\Coroutine::getContext().

📄

global $post;

El post actual en el "Loop". Es mutado constantemente. El "Loop" tradicional (`the_post()`) requiere refactorización para ser aislado por petición HTTP.

El Hito: Compatibilidad con WooCommerce

Lograr que WooCommerce funcione sobre wp-async es la prueba de fuego. WC introduce sus propias complejidades arquitectónicas, principalmente alrededor del manejo de sesiones, carritos y transacciones de base de datos.

🛒 Componentes Críticos a Refactorizar

  • ▪️
    WC_Session_Handler WC utiliza su propio gestor de sesiones guardado en BD. Las escrituras bloqueantes aquí limitan la asincronía. Requiere implementación adaptada a un driver asíncrono en memoria (Redis + Swoole Coroutines).
  • ▪️
    Clase global WC() El helper WC() retorna la única instancia de WooCommerce. Si el carrito se adjunta a esta instancia en Swoole, el usuario A verá el carrito del usuario B. El estado del carrito debe abstraerse al contexto de la petición.
  • ▪️
    Transacciones de Pedidos Durante el checkout, WC hace múltiples queries transaccionales. Es vital asegurar que el connection pool de Swoole no comparta la conexión a BD que está a la mitad de un BEGIN TRANSACTION con otra petición.

Especificación Final wp-async

Para que los plugins existentes sean compatibles con la nueva arquitectura Swoole, se debe implementar una capa de compatibilidad (Adapter Pattern) que intercepte y aisle las operaciones sensibles.

1. Contrato de Aislamiento de Estado

  • Reescritura de add_action y apply_filters para utilizar un almacenamiento basado en el ID de la Corrutina actual.
  • Las variables globales ($_GET, $_POST, $_SERVER) deben ser re-pobladas al inicio de cada evento onRequest de Swoole basándose en el objeto Request de Swoole.

2. Contrato de I/O No Bloqueante

  • El driver de base de datos de WP (wp-db.php) debe ser sustituido por un cliente Swoole MySQL / PDO Asíncrono.
  • Funciones HTTP de WP (wp_remote_get, wp_remote_post) deben sobreescribirse para usar el cliente HTTP asíncrono de Swoole.

3. Contrato de Gestión de Memoria

  • Implementar un proceso de "Garbage Collection" explícito al final del ciclo de vida de la petición asíncrona (Swoole\Http\Response::end()).
  • Vaciar los cachés de objetos locales (WP_Object_Cache) específicos de la petición para evitar fugas.

4. Reglas Estrictas para Plugins

  • Prohibido: Uso de exit() o die(). Esto matará el Worker de Swoole entero. Debe reemplazarse por el lanzamiento de excepciones específicas controladas por el Core.
  • Prohibido: Almacenar datos específicos del usuario en propiedades estáticas de clases.
