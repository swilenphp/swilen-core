WordPress Request Lifecycle

Understanding the traditional request lifecycle is essential before moving WordPress to an asynchronous runtime.

Traditional vs Async Lifecycle

1

Core Bootstrap

Loads WordPress core files. In Swoole this happens only once when the server starts.

2

Plugins

Plugins load and register hooks with the WordPress event system.

3

Init

WordPress parses the request and executes the main database query.

4

Render

The theme template renders HTML and sends the response.

WordPress Hook System

Hooks are the foundation of WordPress extensibility. Plugins interact with the core through actions and filters.

Actions allow plugins to execute code at specific moments during the WordPress lifecycle.

Filters intercept data before it is saved or displayed. They must return a value.

Global State Problem

WordPress relies heavily on global variables. In a persistent runtime like Swoole these globals must be isolated per request.

$wpdb

Must be replaced with a connection pool.

$current_user

Stored inside a coroutine request context.

$post

The WordPress loop must become request-isolated.

WooCommerce Compatibility

WooCommerce introduces sessions, carts and transactions. Making WooCommerce compatible is the ultimate compatibility test.
